  1. **P95 latency tăng đột biến**: `mock_rag.py:23-24` thêm `time.sleep(2.5)` vào hàm `retrieve()`, làm mỗi request chậm thêm 2.5 giây. P95 latency vượt ngưỡng SLO 3000ms.
  2. **Quality score giảm**: `app/agent.py:367-376` — `_heuristic_quality()` giảm 0.2 điểm khi answer chứa `[REDACTED]`, và RAG retrieval chậm làm timeout khả năng tăng.
  3. **Cost tăng**: Thời gian xử lý lâu hơn làm tăng resource consumption, `cost_per_minute_usd` trong `app/metrics.py:135` tăng theo.
  4. **Error rate có thể tăng**: Nếu RAG timeout vượt quá HTTP client timeout (30s trong `scripts/load_test.py`), request sẽ fail và `error_rate_pct` tăng.

- [ROOT_CAUSE_PROVED_BY]: Root cause được chứng minh qua:
  1. **Langfuse traces**: Span `rag-retrieval` trong trace waterfall có duration ~2500ms (so với bình thường < 10ms), trong khi các spans khác (pii-scan, injection-scan, llm-call) giữ nguyên duration. Điều này khoanh vùng bottleneck tại RAG retrieval.
  2. **Log analysis**: Log lines trong `data/logs.jsonl` với `event=response_sent` có `latency_ms` > 3000, correlate với correlation_id của các request chậm.
  3. **Incident state**: `STATE["rag_slow"] = True` trong `app/incidents.py:4` kích hoạt `time.sleep(2.5)` trong `mock_rag.py:23-24` và `app/mock_llm.py:35-36`.
  4. **Metric evidence**: Dashboard Panel 1 (Latency) hiển thị P95 vượt SLO line đỏ, Panel 6 (Quality) hiển thị quality_score_avg giảm.

- [FIX_ACTION]:
  1. Disable incident: `POST /incidents/rag_slow/disable` → `STATE["rag_slow"] = False` trong `app/incidents.py:17-20`
  2. Giảm prompt size: Truncate queries dài > 500 từ trước khi gửi đến RAG
  3. Fallback retrieval source: Khi RAG chính chậm, chuyển sang retrieval source dự phòng (ví dụ: cached responses hoặc local corpus)
  4. Implement timeout: Thêm timeout cho RAG retrieval (ví dụ: 1s), fallback sang empty docs nếu timeout

- [PREVENTIVE_MEASURE]:
  1. **Alerting**: Rule `high_latency_p95` (P2) trong `config/alert_rules.yaml:2-17` cảnh báo khi latency_p95_ms > 5000 for 30m, cho phép phát hiện sớm trước khi ảnh hưởng nhiều users
  2. **Circuit breaker**: Implement circuit breaker pattern cho RAG retrieval — sau N consecutive timeouts, tự động switch sang fallback mode
  3. **RAG performance monitoring**: Theo dõi `rag_docs_count` và retrieval latency trong traces, cảnh báo khi retrieval time > 500ms
  4. **Knowledge base optimization**: Index RAG documents efficiently, implement caching cho queries phổ biến, giảm retrieval time
